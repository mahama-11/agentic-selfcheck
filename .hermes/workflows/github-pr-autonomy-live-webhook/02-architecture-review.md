# Architecture Review — github-pr-autonomy-live-webhook

Role: Architect

## Verdict

**APPROVE WITH REQUIRED LIVE-WEBHOOK CLOSURE WORK.** The existing PR autonomy core is a sound conservative control-plane scaffold, but the live closed loop is not yet complete. The architecture for this requirement should close the gap from external GitHub webhook delivery to deterministic SelfCheck decision and safe GitHub reporter side effects, while preserving the current defaults: allowlisted V repos only, dry-run first, repair disabled unless policy allows, and auto-merge disabled unless separately approved.

## Current architecture baseline

```text
GitHub PR / workflow event
→ Hermes webhook/gateway route (pending live wiring)
→ selfcheck trigger --event github.pull_request --source hermes-webhook
→ events/github-pr-autonomy.yaml route match
→ selfcheck.pr_autonomy.dispatch_payload
→ pr_autonomy_decision
→ reporter action plan / optional GitHub comment + status + merge command
```

Existing implemented pieces:

- `events/github-pr-autonomy.yaml` defines event `github.pull_request`, mode `event`, source allowlist including `hermes-webhook`, `ci-webhook`, and `local`, with debounce and strict audit.
- `selfcheck trigger` attaches `pr_autonomy_decision` for route id `github-pr-autonomy`.
- `selfcheck.pr_autonomy` normalizes GitHub-style PR payloads, applies the V workspace policy, classifies risk, handles missing/pending/failed checks, and returns planned actions.
- `scripts/github_pr_autonomy_report.py` can generate a dry-run report and, with `--apply`, post a PR comment and commit status via `gh`; with `--allow-merge`, it can run merge only for `READY_TO_MERGE` and `MERGE_PR_PLANNED_ONLY`.
- Current policy explicitly covers the 4 V repos: `platform-backend`, `ecommerce-backend`, `platform-frontend`, `ecommerce-frontend`; repair and auto-merge remain disabled.

## Required live webhook closure design

### 1. Hermes webhook route

Required route contract:

```text
POST /<Hermes webhook route for GitHub>
headers:
  X-GitHub-Event
  X-GitHub-Delivery
  X-Hub-Signature-256
body:
  raw GitHub webhook JSON
```

Route behavior must be:

1. Verify HMAC SHA-256 signature using a secret stored outside this repository.
2. Accept only supported event families for this requirement:
   - `pull_request`
   - `workflow_run`
   - optionally `check_run` / `check_suite` if used to refresh check status promptly
3. Normalize delivery into SelfCheck event names:
   - `pull_request` → `github.pull_request`
   - `workflow_run` / check events → either enrich/refresh the matching PR payload and call `github.pull_request`, or add an explicit `github.workflow_run` route that resolves the PR before dispatch.
4. Invoke SelfCheck from `/root/work/agentic-selfcheck` with source `hermes-webhook`:

```bash
python3 -m selfcheck trigger \
  --root /root/work/agentic-selfcheck \
  --event github.pull_request \
  --route github-pr-autonomy \
  --source hermes-webhook \
  --payload-file <redacted-runtime-payload.json> \
  --timeout 300
```

5. Persist only redacted evidence under `reports/github-pr-autonomy-live-webhook/` or `reports/events/`; do not persist raw secrets, Authorization headers, or webhook HMAC material.

Architectural gap: the repository currently contains the SelfCheck event route and a generic Hermes webhook scaffold in `docs/event-callbacks.md`, but no confirmed live Hermes route evidence for this specific PR autonomy flow. Acceptance must not claim live until the external route health check and sample delivery evidence exist.

### 2. GitHub repo webhooks for the 4 V repos

For each allowlisted repository:

- `mahama-11/platform-backend`
- `mahama-11/ecommerce-backend`
- `mahama-11/platform-frontend`
- `mahama-11/ecommerce-frontend`

Create/update a GitHub webhook with:

```text
Payload URL: <Hermes public webhook URL>
Content type: application/json
Secret: stored only in GitHub webhook settings and local Hermes/env secret store
SSL verification: enabled
Events: pull_request, workflow_run; include check_run/check_suite if dispatcher needs direct check refresh
Active: true
```

Required evidence should be generated by GitHub API/CLI without exposing secrets:

```bash
gh api repos/mahama-11/<repo>/hooks --jq '[.[] | {id, active, events, config: {url: .config.url, content_type: .config.content_type, insecure_ssl: .config.insecure_ssl}}]'
```

The output must redact URL tokens if the payload URL contains credentials. It must not print `secret` because GitHub does not return it, and no local env secret should be echoed.

### 3. Event dispatcher contract

The dispatcher should receive an enriched PR snapshot. The current dispatcher requires these fields to make a non-human decision:

- repository owner/name
- PR number/action/state/draft/base/head/head SHA
- `changed_files`
- required check names and statuses in `check_runs`
- labels, if policy/reporting uses them
- optional `repair_attempts`

Live route must therefore either:

1. enrich the raw GitHub webhook payload before calling SelfCheck by fetching PR files and checks, or
2. call a dedicated enrichment command before reporter application.

Without `changed_files`, current policy correctly returns `NEEDS_HUMAN` because risk cannot be classified. Without required checks, it returns `WAITING_FOR_CHECKS`. This is safe but will not satisfy the full autonomous closure until enrichment exists.

`workflow_run` events are not PR payloads; they must resolve associated PRs from `workflow_run.pull_requests[]` or GitHub API, fetch the latest PR head SHA/check context, and dispatch the same PR decision path. Unknown repositories or unsupported actions must remain `IGNORED`/`BLOCKED` per policy.

### 4. Reporter: comment, status, labels

Current reporter supports comment and commit status. Live architecture should make reporter idempotent and include labels:

- **Comment:** update an existing bot-authored marker comment when possible; avoid comment spam on every synchronize/workflow event.
- **Status:** write context `AI Review / PR Autonomy` on the exact evaluated `head_sha`; never report success on stale SHA.
- **Labels:** apply deterministic labels derived from state/risk, for example:
  - `ai:ready-for-human`
  - `ai:needs-human`
  - `ai:blocked`
  - `ai:waiting-checks`
  - `ai:needs-repair`
  - `risk:low|medium|high`

Reporter side effects must be gated by:

```text
policy allows action
AND reporter invoked with explicit apply flag
AND repo is allowlisted
AND payload SHA/PR still matches latest GitHub state
AND GitHub token has minimum required permissions
```

Dry-run must output an action plan containing comment/status/label/repair/merge intentions and target repo/PR/SHA.

### 5. Repair safety gates

The current policy already enforces conservative repair planning:

- repair disabled for all 4 V repos
- `max_repair_attempts: 1`
- denied globs for workflows, migrations, env, secret, credential files
- high-risk PRs route to human review

Live repair executor must add an independent destructive-action gate before creating any external repair job:

```text
decision.state == NEEDS_REPAIR
AND CREATE_REPAIR_DISPATCH in actions
AND repo repair.enabled == true
AND failed checks are repairable
AND changed files all match repair.allowed_globs
AND no denied/high-risk files
AND repair_attempts < max_repair_attempts
AND CLI/runtime flag explicitly enables repair
```

Actual repair execution should remain separate from the dispatcher. Dispatcher decides and records; an executor/owner role performs changes; SelfCheck rerun verifies before any completion or reporter success.

### 6. Auto-merge safety gates

Auto-merge must remain disabled by default. Live architecture may prove capability but must not perform merge unless all gates pass:

```text
policy defaults.auto_merge.enabled == true
AND repo.merge.auto_merge_enabled == true
AND reporter --apply --allow-merge is present
AND decision.state == READY_TO_MERGE
AND MERGE_PR_PLANNED_ONLY in actions
AND risk level is allowed by policy
AND latest GitHub PR head SHA == decision.head_sha
AND required GitHub checks are success on that SHA
AND required SelfCheck feature/verifier groups are PASS/PASS_WITH_NOTES
AND PR is open, non-draft, base branch allowed
AND no human-required condition or high-risk file match
AND branch protection permits squash merge
```

Current V policy keeps clean PRs at `READY_FOR_HUMAN`, so the live rollout can validate merge gating without enabling merge.

### 7. Secret handling

Required secret boundaries:

- Webhook secret lives in GitHub webhook config and Hermes/local env only.
- GitHub token lives in `gh` auth store or environment outside this repo.
- No webhook secret, token, API key, or signed URL may be written to repository files.
- Logs and reports must redact:
  - `Authorization` headers
  - `X-Hub-Signature-256`
  - webhook secret values
  - GitHub tokens
  - payload URLs with embedded credentials
- Raw payload retention should be minimized; if retained for evidence, store sanitized JSON only.

## Key risks and mitigations

| Risk | Impact | Required mitigation |
|---|---:|---|
| Hermes route not actually live | GitHub events never reach SelfCheck | Health check + sample delivery evidence; no false live claim |
| Raw GitHub events lack `changed_files`/checks | Dispatcher safely routes to human/waiting, but closure incomplete | Add enrichment step before dispatcher/reporter apply |
| `workflow_run` does not map directly to PR | CI completion may not update PR decision | Resolve PR via `workflow_run.pull_requests[]` or GitHub API before dispatch |
| Reporter comment spam | Poor PR UX | Upsert marker comment; debounce event route; summarize latest state |
| Status on stale SHA | False green | Re-fetch PR head SHA immediately before status/merge |
| Label drift | Confusing PR state | Remove obsolete `ai:*`/`risk:*` labels before applying current labels |
| Repair or merge side effects too easy to trigger | Unsafe autonomy | Require policy + explicit CLI/runtime flags + latest-state recheck |
| Secret leakage in evidence | Credential compromise | Central redaction and never persist raw secrets |

## Required implementation boundaries for developer role

- Implementation may add Hermes/gateway wiring scripts/config docs, dispatcher enrichment, reporter label support, and evidence commands.
- Implementation must not commit secrets or raw token-bearing outputs.
- Implementation must not enable auto-merge in the V policy unless a separate explicit policy approval exists.
- Implementation must preserve allowlist behavior for only the 4 V repos.
- Implementation should keep SelfCheck generic; V-specific behavior belongs in `pr-autonomy-policies/github-pr-autonomy-v-workspace.yaml` or external deployment configuration, not hardcoded into the core state machine.

## Final architecture decision

Proceed with a live-webhook closure that treats Hermes as the authenticated ingress, SelfCheck as the deterministic dispatcher, and GitHub reporter as a separately gated side-effect adapter. The design is safe to roll out in dry-run/apply-safe mode first. Full acceptance requires concrete redacted evidence for Hermes route health, 4 GitHub repo webhooks, dispatcher decisions, reporter comment/status/labels, validate/audit, and explicit pending evidence for any external access that cannot be completed in the local environment.
